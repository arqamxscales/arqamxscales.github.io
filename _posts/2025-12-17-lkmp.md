## Introduction

In this post, I share my experience participating in the **Linux Kernel Mentorship Program (LKMP)** conducted by the Linux Foundation. The program played a crucial role in shaping my understanding of Linux kernel development and helped me take my first confident steps toward upstream kernel contributions.

---

## Linux Kernel Mentorship Program

The [Linux Kernel Mentorship Program (LKMP)](https://wiki.linuxfoundation.org/lkmp) offers a gateway for enthusiastic developers who want to begin contributing to the Linux kernel—one of the world’s largest and most impactful open-source software projects.  

The program is particularly valuable for beginners who want to understand the Linux kernel development workflow, contribution process, and community culture. It is conducted by **Ms. Shuah Khan**, a veteran kernel developer and subsystem maintainer, along with other experienced mentors.

---

## Applying to the Mentorship

I applied for LKMP during **Fall 2025 (October–December)**. The program runs for three months and is described as *full-time*, not in the sense of excluding a day job, but because of its focused and intensive nature with weekly mentorship sessions.

After applying, I was presented with a list of **prerequisite tasks** designed to prepare candidates for kernel development. These tasks included:

- Compiling the Linux kernel and kernel documentation  
- Fixing basic spelling mistakes and submitting patches for review  
- Researching Linux kernel subsystems  
- Triggering a kernel panic and analyzing stack traces  
- Writing and loading kernel modules  
- Generating patches and submitting them via email  
- Submitting a resume and cover letter  

All tasks had to be completed before a fixed deadline. After successful evaluation, I received an acceptance email confirming my enrollment into the program.  
For more details, refer to the [LKMP wiki](https://wiki.linuxfoundation.org/lkmp).

---

## How the Mentorship Program Works

The program follows a **hands-on, learning-by-doing** approach. Upon enrollment, mentees are provided with a detailed [resource document](https://drive.google.com/file/d/1Nyfdy2OqSjKr0DnlhFtMx4NR6uTs4swq/view) outlining various ways to explore the kernel and identify opportunities for upstream contributions.

A dedicated **Discord group** is created for mentors and mentees. This quickly becomes the most valuable resource—where mentees can observe others’ progress, ask questions, discuss patches, and receive guidance from mentors. The Discord group fosters a strong sense of community and mutual learning.

Every Wednesday from **7:30 PM to 8:30 PM IST**, mentors host office hours via Zoom. These sessions provide an opportunity to ask questions, seek clarification, and discuss challenges faced during development.

To successfully graduate from the program, mentees are expected to upstream **5–10 patches**. Mentees are encouraged to focus on one or two subsystems of interest and identify meaningful contributions, primarily bug fixes.

---

## Ways to Contribute Patches

The resource document shared in the first week outlines multiple ways to contribute to the Linux kernel. I spent nearly a full week studying it before starting my actual work.

Some common contribution paths include:

- Fixing bugs reported by online CI systems  
- Addressing issues reported by **syzbot** (kernel fuzzing tool)  
- Fixing issues identified by **Coverity static analysis**  
- Testing release candidates and reporting or fixing regressions  
- Improving drivers or cleaning up deprecated code  

Although the program emphasizes bug fixes, contributors are not restricted to them—new features and improvements are also welcome, provided the program timeline allows completion of the required number of patches.

---

## My Workflow

I primarily worked inside a **virtual development environment** using Oracle VirtualBox with **Ubuntu 22.04**. I built and ran the kernel using GCC.

In addition, I used **BeagleBone Black** and **TI AM64x EVK** ARM boards for testing. I cross-compiled the kernel using appropriate toolchains and validated fixes on real hardware.

Every Monday, I cloned the latest **Linux release candidate** from Linus’ tree, compiled the kernel and documentation, and ran tests on both my x86 VM and ARM boards. I also enabled several kernel hacking options to catch warnings and debug issues early.

One cardinal rule emphasized throughout the program was:  
> **Every patch must be tested.**

For syzbot-reported bugs, I first validated fixes locally on hardware and then submitted patches to syzbot for automated testing. Only after successful validation did I submit patches upstream.

## My Patches

The list of patches I upstreamed during the mentorship is available here:  
[Upstreamed patches](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/log/?qt=author&q=bhanuseshukumar)

### PCI Subsystem Patches

As part of my day job, I work on PCI device drivers. During the mentorship, the PCI Doorbell feature and its corresponding **kselftests** were upstreamed.

While testing debug kernels and running kselftests on release candidates, I observed a kernel warning related to **sleeping in atomic context**. After analyzing the stack trace, I identified that a mutex was being called from a softirq context.

To fix this, I converted the interrupt handler into a **threaded IRQ** and moved the mutex-related logic into thread context. The patch was tested on TI hardware and successfully upstreamed.

---

### Makefile Patch

While reviewing the kernel `make help` output for documentation targets, I noticed that the output was sorted in a **case-sensitive** manner, making it harder to locate relevant targets.

I fixed this by modifying the Makefile to perform a **case-insensitive sort**, tested it locally, and submitted the patch upstream, where it was accepted.

---

### `net: usb: lan78xx` Driver Bugs

I encountered an **uninitialized memory read** issue in the `lan78xx` driver reported by syzbot. By analyzing the syzbot stack trace, I identified issues in an error-return path.

After fixing the issue, I tested it on hardware and submitted it to syzbot for validation. Once verified, the patch was accepted by the maintainer. During review, I was encouraged to fix similar issues elsewhere in the driver, which I did and submitted as additional patches.

---

### Deprecated Code Fix

Kernel documentation recommends using `kmalloc_array()` instead of `kmalloc()` when allocating arrays to prevent overflow issues. I identified such a case, updated the code accordingly, tested it, and submitted the patch upstream.

---

### DMA-Engine Documentation

While working on DMA-related tasks at my job, I reviewed the **DMA-Engine documentation** and identified areas where API usage could be clarified. I improved the documentation and submitted the changes, which were accepted into the documentation tree.
